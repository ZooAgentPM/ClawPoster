## 设计方向：v2.x 语义搜索系统

### 为什么要做这件事

v1.x 的模板选择逻辑本质上是「从一张固定排序的名单里，从头扫到第一个看起来合适的就用」。这带来了两个问题：

**问题一：越靠前的模板越容易被选中**，和它是否真的适合当前需求无关。名单顺序固定，靠前的模板天然有优势，久而久之，95 套模板里实际被用到的可能只有十来套。

**问题二：模板的描述信息太稀疏**，龙虾没有足够的依据去判断哪个模板更合适。每个模板现在只有二三十字的描述，大量模板看上去描述差不多，龙虾只能靠位置顺序来打破平局。

这两个问题叠加，导致输出的设计图缺乏多样性，用户每次看到的都是那几张脸。

---

### 解决思路：让模板有「语义」

人类选模板的方式不是「从第一个看到最后一个」，而是「脑子里有个印象，然后去找最接近这个印象的」。语义搜索就是在模拟这件事。

**第一步：给每个模板做一张「深度名片」**

现在每个模板的信息只有几十个字。我们要把它扩充成 200 字左右的完整描述，把所有能描述这个模板的信息都写进去：它的视觉风格、配色感觉、情绪调性、适合放什么内容、不适合放什么内容、文字容量有多少。这些信息分别从三个地方来：模板的原始图层数据（字号、颜色、装饰类型）、槽位规格（能放多少字、有几个区块）、以及封面图本身（让 AI 直接看图描述视觉感受）。

信息越丰富，后续匹配就越准。这是整件事最关键的地方。

**第二步：把描述变成「位置」**

有了完整的文字描述，下一步是把它变成一个数字向量——可以理解为在一个多维空间里的一个坐标点。这个转换由 Embedding 模型完成。

这个空间有一个关键性质：**语义相近的模板，坐标点也相近**。「清新手绘风」和「插画轻松风」的坐标会很近；「极简大字冲击」和「红色促销海报」的坐标会很远。

用户的需求描述，也会被转换成同一个空间里的一个坐标点。然后系统只需要计算：哪些模板的坐标和需求的坐标最接近，就返回哪几个。这彻底绕开了排序问题——距离近的自然就是最合适的，和它在名单里排第几无关。

**第三步：确保结果不扎堆**

如果直接返回距离最近的 5 个模板，很可能这 5 个风格几乎一样（比如全是「极简大字」的变体）。所以在出结果前，还要做一轮多样性筛选：从相关的候选里，优先选那些彼此之间差异比较大的，凑成一组风格不雷同的推荐。

---

### 第四步：推荐结果必须附带缩略图

这是整个推荐流程里容易被忽视、但实际上极其关键的一环。

语义搜索再准，返回的如果只是一堆文字描述，龙虾仍然无法做出真正有质量的判断——设计是视觉的，文字永远传达不了「这个模板长什么样」。

所以 `search_templates` 的返回结果必须包含缩略图，让龙虾能直接「看」到候选模板的样子。具体有两种呈现方式：

**方式 A：单张合图**
把推荐的 5 张缩略图拼成一张图，每张下方标注模板编号和 1-2 个关键标签。龙虾收到一张图，扫一眼就能完成视觉比较。好处是信息紧凑，一次返回，适合龙虾自主判断。

**方式 B：多张独立图**
5 张缩略图分别以图片形式返回，每张附带对应的语义说明。好处是龙虾可以逐一解读，描述更灵活。

两种方式不是非此即彼。默认用单张合图（效率高），如果需要针对某个候选深入说明，再单独展示那一张。

**这个设计的根本目的**：推荐结果要让人一眼就能做决策，而不是读完一段话之后还不知道模板长什么样。缩略图是整个选模板体验的核心，不是点缀。

---

### 架构上的一个关键决策：SKILL 只管工具，逻辑全在服务里

Skill 是分发给外部使用的，一旦用户装上去就不会频繁更新。如果选模板的逻辑写在 Skill 里，每次想改进推荐质量都要同步更新 Skill，现实中根本做不到。

所以我们把逻辑放在服务端（MCP server）：Skill 只告诉龙虾「你有哪些工具，调用顺序是什么」，所有的搜索策略、多样性算法、参数调整，全部在服务里做。以后想升级推荐质量，只需要改服务，Skill 不用动。

这是 v2.x 设计上最重要的一个原则。

## v1.10 (2026-03-25)

### 解决的问题
**rough-annotation SVG 装饰丢失**（v1.6 引入的回归 bug）

t708 等含手绘圆圈/高亮装饰的模板，渲染后装饰全部消失。

**根因链路：**
- v1.1 在 DOM 注入路径中加入了"保留 SVG 子节点"逻辑
- v1.6 新增 Vue reactive 注入路径，找到 widget 就 `continue`，完全跳过了 DOM 注入——导致 v1.1 的 SVG 保留逻辑对所有能在 dWidgets 中找到的 widget 全部失效

**修复方案：**
在 Vue reactive 路径入口检查 `widget.text` 是否含 `rough-annotation`；若含，跳过 Vue reactive，fall-through 到 DOM 注入，由 DOM 注入保留 SVG 子节点后再插入新文本。

**影响文件：** render_single.py + render_server.py（两份独立注入代码同步修改）

**回归测试基准：**
- t708（极简冲击）→ 验证 rough-annotation 圆圈/高亮保留，走 DOM 注入路径
- t873（漫画爆款）→ 验证 textEffects 描边颜色正确，走 Vue reactive 路径

### 获得的成果
- t708 等含手绘装饰的模板恢复正常渲染
- 建立了双路径回归测试基准，防止注入逻辑改动再次相互破坏

## v1.9 (2026-03-25)

### 解决的问题
**模板选择缺乏视觉语义信息**：选模板时只有 brief/scenarios/style 等文字描述，无法匹配用户的视觉偏好（如"手绘风""大字霸屏"），也无法按配色、装饰干扰度、版面结构快速过滤。

### 获得的成果
**全部 95 套模板人工标注 4 个视觉元数据字段：**

| 字段 | 说明 | 用途 |
|------|------|------|
| `visual_tags` | 5-6 个中文视觉特征词（如「手绘插画风」「大字霸屏」） | 语义匹配用户视觉偏好 |
| `color_palette` | 主色调列表（中文颜色词） | 配色需求过滤 |
| `decorative_interference` | none / low / medium / high | 内容重时选 low/none |
| `layout_structure` | 6 种固定版面结构（hero_text / split_section 等） | 按版面快速缩小候选 |

数据写入 `template_index_v2.json` 和 `visual_metadata.json`（95 条）。

配套工具：`tools/label_server.py`（port 7003）+ `tools/label.html` 可视化标注界面。

## v1.8 (2026-03-24)

### 解决的问题
**选模板时无法判断模板能否放下内容**：Agent 必须先渲染才能发现溢出，导致同一模板反复迭代。

### 获得的成果
**`template_index_v2.json` 新增 `slot_capacity` 字段（全部 95 套模板）：**

把每个模板所有 `must_edit` 槽位的 `role` / `max_chars` / `max_per_line` / `line_break` 汇总进 template_index，选模板阶段即可做容量粗过滤：

- 内容字数 > `max_chars` × 1.2 → 排除
- 最长行 > `max_per_line` + 2 → 排除

选模板时直接排除明显放不下的模板，大幅减少因溢出导致的重渲次数。

## v1.7 (2026-03-24)

### 解决的问题
1. **渲染速度慢**：每次渲染都新开 Chromium browser，第1次 ~18s，连续渲染无法提速
2. **模板选择缺少背景类型/对齐方式信息**：如无法排除 white_blank 背景、无法知道主标题对齐
3. **溢出预判困难**：Agent 不知道每行能放多少字

### 获得的成果

**新增 `src/render_server.py`（FastAPI HTTP daemon，port 7002）：**
- browser 启动一次，所有请求复用
- 连续渲染第 2 次起从 ~18s 降至 ~5s

**`template_index_v2.json` 新增字段：**
- `background_type`：15 个模板确认（white_minimal / gray_minimal / illustrated_* / photo 等）
- `text_align_main`：71 个模板自动提取主标题对齐方式
- `has_non_replaceable_images`：标注含不可替换示例图的模板

**edit_spec 新增字宽数据（全部 95 套）：**
- `char_px`：单字像素宽
- `element_width_px`：文本框设计宽度
- `max_per_line`：每行最多字数 = `floor(element_width_px / char_px)`

## v1.6 (2026-03-24)

### 解决的问题
1. **textEffects 模板文字颜色渲染错误**：t873（漫画爆款）注入后文字显示为黑色而非橙色+描边
2. **模板库过小**：仅 10 套，覆盖场景有限

**颜色错误根因：**
Playwright 直接操作 DOM 后，Chromium GPU 合成层顺序错乱——描边层（透明填充+黑色描边）渲染在填充层（橙色）上方。

**修复方案：** 优先通过 Vue reactive（`pageGroup[0].dWidgets`）更新 `widget.text`，让 Vue 正常 diff/re-render，绕过 GPU 层问题。

### 获得的成果
- textEffects 模板颜色渲染正确
- **模板库从 10 套扩充至 95 套**：小红书封面（50套）+ 手机海报（15套）+ 小红书配图（20套）
- template_index_v2.json 扩充至 95 条，SKILL.md 新增分类速查表

## v1.2 (2026-03-20)

### 解决的问题
**渲染后无法定位布局问题**：AI 看不到元素的坐标和 UUID 对应关系，无法对重叠/偏移元素进行精确调整。

### 获得的成果
- **Inspect 标注图**：每次渲染自动生成 `t{id}_inspect.png`，彩色边框标注所有元素，左上角显示 UUID 前8位
- **`--adjustments` 参数**：对任意 DOM 元素应用 CSS 样式（transform / opacity / display / fontSize 等），截图前注入
- 渲染器 stdout 改为两行：`RENDER:/path/` + `INSPECT:/path/`
- AI 可自主处理元素重叠、位置偏移等布局问题，无需人工介入

## v1.1 (2026-03-20)

### 解决的问题
**多项渲染注入 bug，导致槽位文字无法正确显示：**

1. **UUID 截断**：edit_specs 中 UUID 为8位前缀，而 DOM ID 是12位，导致槽位注入静默失败
2. **justify 字间距**：`white-space: pre-wrap` + `text-align: justify` 组合下，显式换行前一行被 Chromium 拉伸撑满
3. **rough-annotation SVG 丢失**：注入时清空了 innerHTML，手绘高亮/下划线 SVG 装饰消失
4. **SVG 遮挡**：注入的 span 被 `position: absolute` 的装饰 SVG 盖住

### 获得的成果
- 38 个 UUID 从8位前缀补全为12位（10个模板全部修复）
- `text-align-last: left` 抑制 justify 末行拉伸
- 注入时保留 SVG 子节点（只移除非 SVG 节点）
- 注入 span 设置 `position: relative; z-index: 1` 浮在 SVG 上方
